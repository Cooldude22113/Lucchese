# Lucchese Architecture Summary
_Updated May 2026_

---

## Overview

Lucchese is a local-first, Jarvis-style personal AI operating system built for Alex Hammond. Backend is Python/FastAPI. All routes live in `/routes`. Server entry point is `main.py`. The long-term direction is a persistent cognitive and operational companion — not a chatbot wrapper.

---

## LLM Layer

Dual provider setup. `CHAT_PROVIDER` env var switches between Ollama (local, default) and Claude (Anthropic API).

Local models:
- `gemma2:27b` — fast path
- `qwen2.5:32b` — deep reasoning path

Cloud fallback:
- Claude Sonnet 4.6

All LLM calls route through `routes/chat.py` or `routes/voice.py`.

Known issue: Claude provider does not stream — full response is collected then yielded as a single token.

---

## Memory Infrastructure

Primary store: ChromaDB (persistent, local).

Collections:
- `knowledge` — chunked conversation history
- `facts` — user statements and extracted facts
- `style` — user writing voice
- `documents` — uploaded files
- `summaries` — LLM-compressed category summaries (pipeline partially operational)

Embeddings: `nomic-ai/nomic-embed-text-v1`
Reranking: `cross-encoder/ms-marco-MiniLM-L-6-v2`

All memory logic lives in `routes/memory.py`. `RECENCY_WEIGHT` has been extracted as a tunable constant.

---

## Retrieval Pipeline

Query expansion (2 extra Ollama calls) → multi-collection vector search → cross-encoder rerank → recency scoring → top 10 results injected into system prompt.

Known issues:
- Query expansion runs on every non-web-search message, adding 2 Ollama calls to the hot path.
- Retrieval still prioritizes semantic relevance over recency, operational importance, and current truth.
- Without populated `profile_state`, stale historical memories can dominate context.

---

## Classification Pipeline

Two diverged systems currently exist:

**Live ingest path** (`routes/memory.py`):
- Flat category list
- Applied at ingest time
- Missing fields: `pair_idx`, `user_text_raw`, `routing_done`

**Batch reclassification** (`reclassify.py`):
- Deep tiered taxonomy (9 top-level domains, 3 tiers)
- Ontology, taxonomy, engagement, facts/style extraction, routing metadata
- Applied retroactively to exported ChatGPT/Grok data
- Standalone script — not part of live server

These two systems have diverged schemas and are not compatible without a migration step.

Planned direction: move classification infrastructure to a dedicated second machine to isolate workload from the conversation hot path.

## Persistence

- **SQLite** (`conversations.db`) — conversation history, messages, uploaded document records, active roleplay sessions, and operational state
- **ChromaDB** — all vector/semantic data

---

## Operational State Layer

Technology: SQLite (within `conversations.db` or adjacent)

Managed by: `routes/state.py`, registered in `main.py`, initialized via `init_state_db()` on startup.

Tables:
- `profile_state` — single-row confirmed current truth about Alex (enforced via `CHECK (id = 1)`)
- Projects, tasks, blockers, decisions, daily state

Endpoints:
- `GET /state/profile`
- `PATCH /state/profile` — currently uses replace/clobber semantics; merge semantics review pending (AD-008)

Current limitation: `profile_state` ships empty. Until populated, the model relies entirely on historical ChromaDB retrieval with no confirmed current-truth anchor.

---

## Context Builder

Module: `routes/context_builder.py`

Owns first-pass prompt context assembly. Returns an internal `ContextResult` dataclass. Either tier can fail independently without blocking the other.

Two-tier context model:

**Tier 1 — Current operational truth**
- Source: `profile_state` from SQLite
- Injected first into system prompt
- Treated as ground truth when populated

**Tier 2 — Historical semantic memory**
- Source: ChromaDB collections
- Injected after current facts
- Represents what was true or discussed — not necessarily what is currently true

`chat.py` accepts `ContextResult | None` in `build_system_prompt()`.

Data flow:
```
/chat request
  → build_context(query, conversation_id, mode)
      → get_profile_state() from SQLite
      → search_memory() from ChromaDB
  → build_system_prompt(ContextResult, web_context, sheets_context)
      → Tier 1 current facts injected first
      → Tier 2 historical memory injected second
  → LLM generation
```

Planned expansion: active projects/tasks/blockers injected between Tier 1 and Tier 2.

---

## Profile State Seeding

Module: `seed_profile.py` (standalone script)

Current status: extraction pipeline exists, candidate generation from Chroma facts works, timestamps survive ingestion. Manual confirmation workflow exists conceptually.

Known limitations:
- Extraction quality still unreliable
- Semantic recency does not equal current truth
- Imported historical memories may contain superseded identity/state information

Confirmed direction (AD-007, AD-009): historical memories must not automatically become current profile truth. Extracted candidates require human confirmation before promotion to `profile_state`.

---

## Summaries Layer

Collection exists. `summarize.py` exists for category summaries after reclassification. Generation pipeline needs confirmed operational use and testing.

---

## Voice Pipeline

Current capabilities:
- STT via Whisper (tiny model)
- TTS via ElevenLabs
- Streaming voice responses

Known gaps:
- No VAD
- No interruption handling
- No persistent session mode
- No voice-aware formatting
- No latency optimization

Long-term direction: voice-first interaction mode (AD-005).

## Business Logic Modules

- `routes/deal.py` — pure Python BTL property deal analyser, no LLM
- `routes/shopify.py` — creates Shopify products from Google Sheets recipe data
- `routes/scrape.py` — scrapes websites, builds structured review prompt for LLM
- `routes/roleplay.py` — Carol persona for property pitch practice, session-based, triggered via chat intercept (no TTL on sessions)
- `routes/voice.py` — STT and TTS handling

## Ingestion Scripts

Root-level standalone scripts (not part of live server):
- `ingestgpt.py`, `grok.py`, and versioned variants
- Bulk-load exported conversation history into ChromaDB

---

## Infrastructure Direction

**Main machine:**
- Conversation models (Ollama)
- LLM generation
- Voice interaction
- FastAPI server

**Second machine (planned, AD-001):**
- ChromaDB HTTP server
- Embedding generation
- Classification model
- Memory infrastructure

Purpose: isolate memory workload, reduce hot-path GPU contention, improve scalability. Second machine becomes a dependency — graceful degradation required.

| Issue | Status |
|---|---|
| `profile_state` ships empty — historical Chroma memories dominate context | Unresolved — seeding pipeline exists but not operationalized |
| `PATCH /state/profile` clobber semantics — omitted fields become NULL | Pending decision (AD-008) |
| Live ingest and reclassify.py schema divergence | Unresolved — migration required |

| Query expansion adds 2 Ollama calls to every hot-path message | Known, unresolved |
| Roleplay sessions have no TTL | Known, unresolved |
