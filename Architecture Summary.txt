# Lucchese Architecture Summary
_Updated May 2026_

---

## Overview

Lucchese is a local-first, Jarvis-style personal AI operating system built for Alex Hammond. Backend is Python/FastAPI. All routes live in `/routes`. Server entry point is `main.py`. The long-term direction is a persistent cognitive and operational companion — not a chatbot wrapper.

**Current phase: Stabilize.** The architecture is conceptually ahead of its operational maturity. Feature expansion is explicitly deferred. The active work is hardening what exists, not building new capabilities. Any Claude project working on Lucchese should internalize this constraint before touching anything.

---

## Development Pipeline

Lucchese uses a formal five-stage AI engineering pipeline (AD-010):

| Stage | Role |
|---|---|
| **Lucchese Main** | Operational oversight, roadmap, blocker identification |
| **Architecture & Design** | Technical specifications, implementation-safe blueprints |
| **Implementation** | Safe application of approved specs into the codebase |
| **Review** | Correctness audit, production readiness, architectural compliance |
| **Stabilize** | Operational hardening, observability, debt reduction, deployment discipline |

**Stabilize is the currently active stage.** Work outside the allowed stabilization scope is forbidden. No new features, no infrastructure redesigns, no speculative systems.

## LLM Layer

Dual provider setup. `CHAT_PROVIDER` env var switches between Ollama (local, default) and Claude (Anthropic API).

Local models:
- `gemma2:27b` — fast path
- `qwen2.5:32b` — deep reasoning path

Cloud fallback:
- Claude Sonnet 4.6

Known issue: Claude provider does not stream — full response is collected then yielded as a single token.

---

## chat.py — Highest-Priority Structural Risk

**`chat.py` is a god-route orchestrator and must not receive any new responsibility.**

It currently handles: routing, web search, memory commands, scraping, roleplay, LLM calls, streaming, ingestion, and prompt construction within a single hot path. This accumulation of responsibility was confirmed by the external audit as the most likely first failure point under real operational stress. God-route orchestration degrades gradually and fails suddenly.

The correct extraction path is `routes/context_builder.py`. That decomposition work must continue and accelerate. Any change that adds responsibility to `chat.py` is an architectural regression.

**Restriction (AD-011): No new subsystems are to be routed through `chat.py`.**

## Memory Infrastructure

Primary store: ChromaDB (persistent, local).

Collections:
- `knowledge` — chunked conversation history
- `facts` — user statements and extracted facts
- `style` — user writing voice
- `documents` — uploaded files
- `summaries` — LLM-compressed category summaries (**experimental; not confirmed operational in live retrieval**)

Embeddings: `nomic-ai/nomic-embed-text-v1`
Reranking: `cross-encoder/ms-marco-MiniLM-L-6-v2`

All memory logic lives in `routes/memory.py`. `RECENCY_WEIGHT` has been extracted as a tunable constant.

---

## Retrieval Pipeline

Query expansion (2 extra Ollama calls) → multi-collection vector search → cross-encoder rerank → recency scoring → top 10 results injected into system prompt.

Known issues:
- Query expansion runs on every non-web-search message, adding 2 Ollama calls to the hot path with no clearly bounded behavior.
- Retrieval prioritizes semantic relevance over recency, operational importance, and current truth.
- Without populated `profile_state`, stale historical memories dominate context. This is the current default operational state.

---

## Classification Pipeline

Two diverged systems currently exist:

**Live ingest path** (`routes/memory.py`):
- Flat category list
- Applied at ingest time
- Missing fields: `pair_idx`, `user_text_raw`, `routing_done`

**Batch reclassification** (`reclassify.py`):
- Deep tiered taxonomy (9 top-level domains, 3 tiers)
- Ontology, taxonomy, engagement, facts/style extraction, routing metadata
- Applied retroactively to exported ChatGPT/Grok data
- Standalone script — not part of live server

**Active risk (AD-002, AD-016):** These two systems have diverged schemas. Metadata written during live ingestion may not match what reclassification and retrieval layers expect. This is a silent failure class — it degrades retrieval quality gradually with no visible error. Schema unification is a stabilization priority; no tooling currently exists to detect the drift.

**Planned direction:** Move classification infrastructure to a dedicated second machine. **This migration is deferred** — see Infrastructure Direction below.

## Operational State Layer

Technology: SQLite

Managed by: `routes/state.py`, registered in `main.py`, initialized via `init_state_db()` on startup.

Tables:
- `profile_state` — single-row confirmed current truth about Alex (enforced via `CHECK (id = 1)`)
- Projects, tasks, blockers, decisions, daily state

Endpoints:
- `GET /state/profile`
- `PATCH /state/profile` — **uses merge semantics** (resolved, AD-008). Omitted fields are preserved from the existing row. Explicit field clearing via `clear_fields: list[str]` parameter.

**Critical limitation:** `profile_state` ships empty and remains empty unless manually populated or seeded via `seed_profile.py` with human confirmation. Until populated, the model receives no confirmed current-truth anchor and defaults entirely to historical ChromaDB retrieval. This means the model may confidently present outdated information — old projects, old age, inactive goals — as present reality. This is not a theoretical risk. It is the current default behavior.

**Known risk (AD-003, AD-013 — partially resolved):** Startup validation is now deployed (STAB-005). On every process start, `routes/startup_validator.py` confirms required tables are present and correctly structured, validates the environment, and checks ChromaDB connectivity. Hard-stop on missing tables or invalid configuration. AD-013 satisfied.

Remaining gap: no defined rollback procedure if the state layer is corrupted or becomes inconsistent between process starts. The state layer holds the Tier 1 ground truth the entire context pipeline depends on — rollback discipline remains an open operational risk.

## Context Builder

Module: `routes/context_builder.py`

Owns first-pass prompt context assembly. Returns an internal `ContextResult` dataclass. Either tier can fail independently without blocking the other. This is the correct architectural direction for extracting responsibility from `chat.py`.

Two-tier context model:

**Tier 1 — Current operational truth**
- Source: `profile_state` from SQLite
- Injected first into system prompt
- Treated as ground truth when populated
- **Currently empty in the live system**

**Tier 2 — Historical semantic memory**
- Source: ChromaDB collections
- Injected after current facts
- Represents what was true or discussed — not necessarily what is currently true

`chat.py` accepts `ContextResult | None` in `build_system_prompt()`.
`voice.py` uses `build_context()` — same tier contract as chat.

Data flow:
```
/chat request
  → build_context(query)
      → get_profile_state() from SQLite
      → search_memory() from ChromaDB
  → build_system_prompt(ContextResult, web_context, sheets_context)
      → Tier 1 current facts injected first
      → Tier 2 historical memory injected second
  → LLM generation
```

**Known risk (AD-006 — partially resolved):** Tier failures silently degrade to empty strings to keep the chat route alive. Context assembly observability is now deployed (STAB-001) — `emit_context_log()` emits a structured entry on every main-path inference recording which tiers were populated, which failed, and char counts. Tier failures now surface as WARNING entries before user-visible degradation. The silent failure mode is now observable; the degradation policy itself (empty string fallback) remains unchanged.

**Known risk (AD-015):** `build_system_prompt()` has no enforced token budget and no explicit priority ordering beyond Tier 1 before Tier 2. Uncontrolled context growth as additional sources are added is a known failure mode. Token budget and priority enforcement (steps 2–6 below) are not yet implemented:

1. Confirmed profile state
2. Active projects / tasks / blockers
3. Recent decisions / daily state
4. Relevant summaries
5. Semantic memories
6. Recent conversation

Planned expansion: active projects/tasks/blockers injected between Tier 1 and Tier 2, after profile state seeding is operationalized.

## Profile State Seeding

Module: `seed_profile.py` (standalone script)

Current status: extraction pipeline exists, candidate generation from Chroma facts works, timestamps survive ingestion. Manual confirmation workflow exists conceptually.

Known limitations:
- Extraction quality still unreliable
- Semantic recency does not equal current truth
- Imported historical memories may contain superseded identity and project information

**Confirmed direction (AD-007, AD-009):** Historical memories must not automatically become current profile truth. Extracted candidates require human confirmation before promotion to `profile_state`. This constraint is non-negotiable — `profile_state` is injected as authoritative ground truth at every inference.

## Observability — Partially Implemented

**Observability is not a future concern. It is a current blocker for production maturity (AD-012).**

Deployed:
- Inference-time context assembly logging via `lucchese.context` stream (STAB-001)
- `tier1_status`, `tier2_status`, `total_assembled_char_count` per inference (STAB-001)
- Log rotation on `lucchese.context` — 5 MB cap, 5 backups (STAB-003)
- Startup validation logging via `lucchese.startup` stream (STAB-005)
- Environment, SQLite, schema, profile_state row, and ChromaDB checks at every process start (STAB-005)

Remaining gaps:
- No retrieval inspection output — ChromaDB results, rerank scores, recency weights not logged
- No centralized telemetry or structured run trace
- Intercept paths in `chat.py` (action plan, scrape) produce no observability entries
- `lucchese.startup` logger has no file handler (TD-004, tracked)

**Minimum required before production maturity can be claimed:**
- ~~Structured logging of context tier state at inference time~~ — **DONE (STAB-001)**
- ~~Explicit failure signals from context assembly~~ — **DONE (STAB-001)**
- ~~Startup validation with dependency and environment checks~~ — **DONE (STAB-005)**
- ~~Configuration validation on startup~~ — **DONE (STAB-005)**
- Retrieval inspection output — **NOT YET IMPLEMENTED**

## Summaries Layer

Collection exists in ChromaDB. `summarize.py` exists for category summaries after reclassification. **Generation pipeline has not been confirmed as operationally integrated or tested in live use.** Do not treat the summaries collection as a reliable retrieval source.

## Voice Pipeline

Current capabilities:
- STT via Whisper (tiny model)
- TTS via ElevenLabs
- Streaming voice responses
- `voice.py` correctly uses `build_context()` — Tier 1 and Tier 2 injected via context_builder (do not regress this)

Known gaps:
- No VAD
- No interruption handling
- No persistent session mode
- No TTL or lifecycle management for voice sessions
- No voice-aware response formatting
- No latency optimization

**Voice feature expansion is suspended (AD-005).** The existing integration is sound and must not be regressed. No new voice features are to be added until `chat.py` decomposition, context observability, and startup validation are complete.

## Infrastructure Direction

**Main machine (current):**
- Conversation models (Ollama)
- LLM generation
- Voice interaction
- FastAPI server

**Second machine (AD-001) — Confirmed Direction, Deliberately Deferred:**
- ChromaDB HTTP server
- Embedding generation
- Classification model
- Memory infrastructure

**This migration is not active.** The second machine introduces a hard runtime dependency that requires graceful degradation design which does not yet exist. Migrating before startup validation, failure signaling, and context-assembly observability are in place compounds operational risk. This work does not proceed until `chat.py` decomposition and core stabilization are complete.
